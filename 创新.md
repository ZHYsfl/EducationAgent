# 1.ChannelBuffer Auto-tuning(通道缓冲区自适应机制)
- 位置: internal/adaptive/adaptive.go
- 机制:
 - 监控6个Go chanel的利用率: audio_ch, asr_audio_ch, asr _result_ch, sentence_ch, write_ch, tts_chunk_ch
 - 根据峰值长度和阻塞次数动态调整缓冲区大小
 - 利用率>80%或有阻塞 -> 扩大1.5倍
 - 利用率<20%且无阻塞 -> 缩小0.75倍
 - 每个通道有min/max限制(如audio_ch: 50-800)
- 创新点:运行时自适应调整内存占用，避免channel阻塞导致的延迟

# 2.动态分层system Prompt(4层架构)
- 位置: agent/context_engine.go 的 buildsystemPrompt()
- 机制:
 - Layer1:角色定位(动态)
  - 需求收集阶段:显示"还需要收集xxx"+"己收集xxx”
  - 收集完成后:切换为固定角色描述
 - Layer2:任务列表(动态)
  - 有任务时出现，显示所有task_id和当前任务
  - 多任务时加提示"请先确认用户指的是哪一个任务"
- Layer 3:待解决冲突(动态)
  - 有冲突时出现，显示context_id和question
  - 冲突解决后自动消失
- Layer5:协议指令(静态)
 - 工具使用规范，始终存在
- 创新点:System prompt随会话状态实时更新，模型始终看到最新上下文

# 3.异步工具与Idle Trigger机制
- 位置: agent/context_engine.go 的 Enqueuecontext() 和 flushToolResults()
- 机制:
 - 工具结果通过Enqueuecontext()进入contextQueue
 - 特殊事件立即处理:update_requirements,conflict_question, task_list_update
 - 其他消息进队列，通过flushToolResults()批量drain到历史
 - Idle Trigger:如果session是idle状态，工具结果到达时自动触发 startProcessing(sctx,"")(userText为空)
- 创新点:工具结果可以在用户未发言时主动触发AI播报，实现真正的异步交互

# 4.三路合并冲突解决协议
- 位置:agent/session.go + agent/http.go + agent/context_engine.go
- 机制:
 - 冲突注入:后端通过conflict_question 事件调用 AddPendingQuestion()
 - 冲突展示:System Prompt Layer 3动态显示未解决冲突
 - 冲突解决:AI调用@{ppt mod} -> 后端处理 -> 返回 ppt_ mod_result事件(带 conflict_resolved:true) -》 调用 ResolvePendingQuestion()清理
 - 状态同步:冲突解决后下一轮systemprompt自动移除该冲突
- 创新点:PPT生成的三路合并冲突(用户意图、AI理解、渲染结果)通过对话式交互解决，而非传统的diff工具

# 5.多Goroutine事件驱动Pipeline
- 位置: agent/pipeline_process.go, agent/pipeline_think.go, agent/session.go
- 架构:
 - ASRLoop:asrLoop()goroutine，持续接收音频并识别
 - Think Loop: thinkLoop() goroutine + 多个 runThinkStream() goroutines, KV cache预热
 - Process Flow:每次 startProcessing()在新goroutine中执行
 - TTS Worker:ttsWorker()goroutine，流式合成语音
 - Action Executors:每个 executor.Execute()在独立goroutine中异步执行
- 创新点:不是简单的O/T/A三线程，而是事件驱动的多goroutine pipeline，各模块异步并发，最大化吞吐

# 6.打断恢复与不完整动作处理
- 位置: pipeline.go 的 OnInterrupt()
- 机制:
 - 用户打断时，等待最多2秒让未闭合的 @{...} 完成
 - 如果2秒后仍未闭合，删除不完整的 @{ 前缀
 - 保存已输出内容到历史，末尾添加 </ interrupted >标记下次推理时模型可以看到</ interrupted >并恢复上下文
- 创新点:流式输出中的工具调用可以被优雅打断，保留可恢复的上下文轨迹

# 7.KV Cache预热(Think Loop)
- 位置:pipeline_think.go 的thinkLoop()和 runThinkstream()
- 机制:
 - ASR每产生7个字符增量，触发一次thinkstream
 - 用部分ASR结果调用LLM，所有生成token丢弃
 - 目的是预热KV cache，等VAD结束时cache已就绪
 - 正式推理时可以直接复用cache，减少首token延迟
- 创新点:利用ASR流式特性提前预热模型，降低用户感知延迟
