# 智能语音辅助教学课件生成系统 — 特色综述

本系统是一款面向教师授课场景的**智能语音辅助教学课件生成系统**，以语音交互为核心驱动，融合了大语言模型（LLM）、检索增强生成（RAG）、知识图谱、多智能体协作等前沿技术。用户可以通过自然语音对话的方式，完成 PPT 课件的智能生成、修改与迭代，全程无需键盘操作，系统在"听"与"说"之间完成教学需求的理解、内容的生成、冲突的解决。以下从**系统架构**、**数据与模型**、**核心算法与机制**三个层面，系统性地阐述本项目的技术特色与创新。

## 3.1 系统架构

### 3.1.1 微服务架构总览

本系统采用 **Go 微服务架构**，将功能拆分为多个独立部署、独立扩展的服务，各服务通过 HTTP/WebSocket 进行通信。整个系统由以下五个核心服务组成：

| 服务名称 | 技术栈 | 核心职责 |
|---|---|---|
| **voice_agent_v2** | Go (Gin + Goroutine) | 语音交互核心：ASR → LLM → TTS，管道编排与协议解析 |
| **memory_service** | Go (Gin + Redis + PostgreSQL) | 用户记忆管理：长期偏好、工作记忆、召回策略 |
| **kb-service** | Go (Gin + Qdrant + PostgreSQL) | 知识库服务：文档解析、Embedding、RAG 检索、重排序 |
| **zcxppt** | Go (Gin + Python 渲染) | PPT 生成：LLM 生成页面代码、Python-pptx 渲染、三路合并 |
| **interface** | Go (Gin) | API 网关：认证、OSS 代理、搜索聚合、回调通知 |

![系统架构图](E:/whole/assets/system_architecture.png)

图 1 展示了五个服务之间的调用关系。`voice_agent_v2` 作为**编排中枢**，协调其他四个服务：用户对话由 Voice Agent 处理，涉及知识查询时调用 `kb-service`；生成 PPT 时调用 `zcxppt`；涉及记忆召回时调用 `memory_service`；所有对外 HTTP 请求经 `interface` 网关统一转发。各服务之间完全解耦，可以独立开发、测试与部署。

### 3.1.2 多 Goroutine 事件驱动 Pipeline

传统的语音助手通常采用 O（Overseer）/T（Thinker）/A（Actor）三线程串行模型，即一条语音输入经过 ASR 识别、LLM 推理、TTS 合成三个顺序阶段。本系统在 `voice_agent_v2` 中实现了**多 Goroutine 事件驱动 Pipeline**，将各环节彻底解耦为独立并发的执行单元，通过 Channel 进行事件驱动通信。

```
┌─────────────────────────────────────────────────────────────────┐
│                    StartInteractive (Goroutine 启动入口)          │
│                                                                  │
│  ┌─────────────────┐           ┌─────────────────┐             │
│  │   asrLoop       │ ─userInputCh─→ │ thinkLoop      │             │
│  │  (Goroutine)    │           │  (Goroutine)    │             │
│  │  音频采集        │           │  KV Cache 预热   │             │
│  │  ASR 识别       │           │  (丢弃 token)    │             │
│  └────────┬────────┘           └────────┬────────┘             │
│           │ audioCh                        │                       │
│           │ VAD End Ch                     │                       │
└───────────┼────────────────────────────────┼────────────────────┘
            │                                │
            ▼                                ▼
  ┌─────────────────────────────────────────────────────┐
  │              startProcessing (per-call)             │
  │                                                      │
  │   tokenCh (LLM 流式输出) ──────────────────────────│
  │        │                                           │
  │        ├── executeAction() ──→ sentenceCh          │
  │        │  (独立 Goroutine via Executor)            │
  │        │                                           │
  │        └── sentenceCh ──────────────→ ttsWorker   │
  │                                        (Goroutine)  │
  └─────────────────────────────────────────────────────┘
```

该 Pipeline 的核心设计包含以下模块：

**ASR Loop（asrLoop）**：持续接收音频数据，调用 ASR 服务进行语音识别。对于部分识别结果（Partial），实时通过 `userInputCh` 发送给 Think Loop；对于最终识别结果，触发 Process Flow。VAD（Voice Activity Detection）结束信号触发最终结果的处理流程。

**Think Loop（thinkLoop）**：监听 `userInputCh`，接收 ASR 发送的增量文本。每当用户输入增加至少 7 个字符（可配置），触发一次 `runThinkStream`，调用 LLM 生成 token 但**丢弃所有输出**。其目的是利用 ASR 流式特性，在用户说话过程中提前预热 LLM 的 KV Cache，使得 VAD 结束时正式推理可以直接复用已就绪的 Cache，将首 token 延迟降低到用户几乎无感知。

**Process Flow（startProcessing）**：在独立 Goroutine 中执行。负责调用 LLM 进行正式推理、解析协议动作（`@{...}` 格式）、过滤可见文本（`#{...}` 内部思考不显示）、分句并发送到 TTS Worker。Token 超出预算时自动注入填充语（Filler Phrase），保持对话流畅自然。

**TTS Worker（ttsWorker）**：监听 `sentenceCh`，接收完整句子。调用 TTS 服务流式合成语音，每合成一个音频块立即通过 WebSocket 发送给前端，实现边合成边播放的低延迟体验。

**Action Executor（executeAction）**：每个动作（`@{kb_query}`、`@{ppt_mod}` 等）在独立 Goroutine 中异步执行，执行完成后通过 `EnqueueContext` 回调，将结果注入工具结果队列。整个过程中 LLM 生成动作后**不等待结果立即继续**，最大化并发吞吐量。

### 3.1.3 异步工具与 Idle Trigger 机制

工具执行结果（如知识库查询结果、PPT 生成状态）通过 `EnqueueContext` 进入工具结果队列（`contextQueue`），通过 `flushToolResults` 在下一轮 LLM 调用前**一次性批量 drain** 到对话历史中。这一设计避免了逐个等待工具结果的低效，也避免了频繁切换 LLM 调用造成的延迟累积。

Idle Trigger 是本系统的另一项重要创新。当工具结果到达时，如果会话处于 `StateIdle` 状态（用户未发言、无正在进行的推理），系统通过 `CompareAndSetState` 原子操作将状态切换为 `StateProcessing`，自动触发 `startProcessing`——即**在用户未主动说话的情况下，工具结果可以主动触发 AI 播报**。这一机制使得异步工具调用成为可能：LLM 调用知识库后无需等待结果返回，可以继续生成其他内容；知识库结果返回后，Idle Trigger 自动唤醒 AI 继续处理并播报结果，实现真正的异步交互体验。

![Idle Trigger 触发流程图](E:/whole/assets/idle_trigger_flow.png)

### 3.1.4 会话状态机

会话在 `Idle`、`Listening`、`Processing`、`Speaking` 四种状态之间流转，由原子 CAS（Compare-and-Swap）操作保证状态转换的线程安全：

- **StateIdle**：系统空闲，等待用户输入。
- **StateListening**：用户正在说话，ASR Loop 持续接收音频并进行语音识别。
- **StateProcessing**：LLM 正在推理，token 正在生成。
- **StateSpeaking**：TTS 正在播报，句子已分句发送至 TTS Worker。

用户在任何时刻的 VAD 开始信号（开始说话）都会触发 `OnInterrupt`：取消当前的 Think/Process 流、等待最多 2 秒让未闭合的协议块完成、裁剪不完整的 `@{...}` 前缀、保存已输出内容并在末尾追加 `</interrupted>` 标记。下一轮推理时，LLM 可以看到 `</interrupted>` 并据此恢复上下文。

## 3.2 数据与模型

### 3.2.1 记忆服务的多维度召回策略

`memory_service` 是系统的记忆中枢，采用**分层多维度评分召回**策略，而非简单的向量相似度检索。整个召回流程包含四个关键阶段：

**阶段一：意图检测**。系统通过检测用户查询中的关键词，自动识别四种意图类型：

| 意图类型 | 检测关键词 | 召回偏向 |
|---|---|---|
| **Continuation** | continue、接着、之前、pick up | 优先同 session 的历史记忆 |
| **RequirementGather** | 知识点、教学目标、目标受众 | 收集缺失的字段信息 |
| **PreferenceFocus** | 偏好、风格、配色、字体 | 长期偏好记忆 |
| **CurrentLessonOrTask** | 本节课、这节课、当前任务 | 当前课程相关记忆 |

**阶段二：多维度评分**。每个候选记忆条目在以下维度获得加权分数：

| 评分维度 | 最高加分 | 说明 |
|---|---|---|
| 关键短语匹配 | +12 | query 精确匹配记忆的 key |
| 值匹配 | +6 | query 匹配记忆的 value |
| Token 匹配（key/value/context） | +7 / +4 / +2 | 多 token 累积加分 |
| 工作记忆信号匹配 | +6 | 与当前对话中的关键词重叠 |
| Session 来源匹配 | +3 / +1 | 当前 session 或其他 session |
| 时间衰减奖励 | +2 / +1 | 7 天内 / 30 天内 |

**阶段三：智能预算分配**。根据意图类型动态分配 Facts 和 Preferences 的召回比例：

- **默认**：50% Facts、50% Preferences
- **偏好聚焦**（PreferenceFocus）：70% Preferences、30% Facts
- **需求收集 / 继续 / 当前任务**：65% Facts、35% Preferences

**阶段四：偏好衰减机制**。长期偏好（Standing Preference）每 30 天衰减 10%（幂函数衰减），置信度低于 0.3 的条目前缀标记 `[低置信度]`。同时，系统区分"长期习惯偏好"和"任务范围偏好"——后者（如"这节课用蓝色"）不会写入持久存储，避免将一次性偏好固化为长期记忆。

### 3.2.2 打断检测模型（LoRA 微调）

`E:\model` 目录下的打断检测模型基于 **Qwen3-0.6B** 进行 LoRA 微调，专门解决语音助手场景下的用户打断检测问题。训练数据为 5000 条（含 CoT 推理路径），通过构造"对话历史 + 当前语音片段 → 是否应打断"的二分类任务，使模型学会判断当前 ASR 结果是否代表了用户的主动打断意图。

| 配置项 | 值 |
|---|---|
| 基础模型 | Qwen3-0.6B（6 亿参数） |
| LoRA rank (r) | 8 |
| LoRA alpha | 16 |
| 学习率 | 1e-4 |
| Batch Size | 2（梯度累积 8 步） |
| Epochs | 3 |
| 训练集 / 验证集 / 测试集 | 4000 / 500 / 500 |

测试集性能达到：**准确率 91.2%，精确率 97.37%，召回率 90.49%，F1 分数 0.938，AUC-ROC 0.98**。

该模型在 Pipeline 中驱动 `OnInterrupt` 的决策：当模型判定当前语音片段为打断意图时，立即触发 `onVADStart`，取消当前的 LLM 推理和 TTS 播报，切换到聆听状态。

### 3.2.3 知识库的 RAG 检索流程

`kb-service` 实现了完整的 RAG（检索增强生成）流程，支持多格式文档的解析、Embedding、向量存储与重排序。整体流程分为**索引阶段**和**检索阶段**：

- **索引阶段**：文档上传后，首先通过多格式解析器提取文本内容，然后按 800 字 / 100 字重叠的分块策略进行文本切分，最后通过 BGE-M3 模型生成 1024 维的 Embedding 向量，写入 Qdrant 向量数据库。

- **检索阶段**：用户查询到来后，首先由 LLM 对模糊查询进行意图精化（Query Refinement），扩写为精准的中文检索关键词；随后并行执行 Dense 向量检索（Qdrant）和 BM25 关键词检索，两路结果通过 RRF（Reciprocal Rank Fusion）算法进行融合排序；最后由 Jina Reranker 对 Top-N 结果进行精细重排序，返回最终结果。

**混合检索**是本系统知识库的核心创新：Dense 向量检索捕捉语义相似性，BM25 关键词检索保障精确匹配，两者的结果通过 **RRF（Reciprocal Rank Fusion）** 算法融合：

```
RRF(d) = Σ weight_i / (k + rank_i(d))
```

其中 k=60，dense 权重默认为 0.5，可通过 API 参数调整。Jina Reranker 在融合后对 Top-N 结果进行精细重排序，进一步提升相关性。

**LLM 意图精化**（Query Refinement）将用户模糊的自然语言查询扩写为精准的中文检索关键词：

- 输入：`"帮我找找那个三角形的东西"`
- 输出：`"三角形定义 三角形内角和 三角形面积公式 几何图形"`

当 LLM 服务不可用时，系统自动降级为原始查询，确保检索链路始终可用。

### 3.2.4 冲突检测的 LLM Tool Calling 机制

`kb-service` 中的冲突检测（Conflict Detection）完全由 **LLM 通过 Tool Calling 动态判断**，不依赖任何硬编码规则：

```go
// LLM 收到两条指令后，通过 report_conflict_result tool 结构化输出判断结果
report_conflict_result(is_conflict: boolean, reason: string, suggestion: string)
```

这一设计使得冲突检测具有极强的泛化能力——无论冲突的语义多么复杂、多么出人意料，LLM 都能基于对指令的理解做出判断。当 LLM 服务不可用时，降级策略返回 `IsConflict=false`（不误判）。

## 3.3 核心算法与机制

### 3.3.1 Channel Buffer 自适应机制（Auto-tuning）

`internal/adaptive/adaptive.go` 实现了一套基于利用率的动态通道缓冲区调整系统。系统持续监控 6 个关键 Go Channel 的使用情况：

| 通道名称 | 用途 | 最小值 | 最大值 | 默认值 |
|---|---|---|---|---|
| `audio_ch` | 音频数据流 | 50 | 800 | 200 |
| `asr_audio_ch` | ASR 音频输入 | 5 | 80 | 20 |
| `asr_result_ch` | ASR 识别结果 | 5 | 80 | 20 |
| `sentence_ch` | 句子分句输出 | 5 | 80 | 20 |
| `write_ch` | WebSocket 写队列 | 64 | 1024 | 256 |
| `tts_chunk_ch` | TTS 音频分块 | 5 | 80 | 20 |

**调整规则**：

- **扩容触发**：利用率 > 80% **或** 发送阻塞次数 > 0 → 扩大 **1.5 倍**（向上取整）
- **缩容触发**：利用率 < 20% **且** 发送阻塞次数 = 0 → 缩小 **0.75 倍**（向下取整）
- **边界约束**：调整结果始终受到 min/max 限制的约束

```
PeakLen = 周期内 channel 的最大长度（峰值）
利用率 = PeakLen / 当前容量
SendBlocks = 周期内发送被阻塞的次数
```

每轮调整后自动重置 metrics，开始新一轮监控。调整后的配置可持久化到 JSON 文件，实现重启后恢复。该机制的核心创新在于：**运行时自适应调整内存占用**，避免固定 buffer 大小在高负载时发生阻塞、或在低负载时浪费内存。

### 3.3.2 动态分层 System Prompt（4 层架构）

`agent/context_engine.go` 的 `buildSystemPrompt()` 实现了**随会话状态实时更新的动态分层 System Prompt**，模型在每一轮推理时都能看到最新的上下文：

| 层次 | 名称 | 性质 | 内容 |
|---|---|---|---|
| **Layer 1** | 角色定位 | 动态 | 需求收集阶段显示"还需收集 xxx"+"已收集 xxx"；收集完成后切换为固定角色描述 |
| **Layer 2** | 任务列表 | 动态 | 所有 task_id 和当前任务；多任务时提示"请先确认用户指的是哪一个" |
| **Layer 3** | 待解决冲突 | 动态 | 所有 context_id、task_id 和冲突问题；冲突解决后自动从 Layer 3 消失 |
| **Layer 4** | 协议指令 | 静态 | 工具使用规范，始终存在 |

Layer 3 的动态消失机制是本设计的亮点：冲突通过后端 `conflict_question` 事件注入 `PendingQuestions` map → 写入 Layer 3 → 用户通过 `ppt_mod` 反馈 → 后端处理后返回 `ppt_mod_result` 事件 → 调用 `ResolvePendingQuestion()` 从 map 中删除 → 下一轮 System Prompt 自动不再显示该冲突。整个过程无需显式刷新，状态驱动更新。

### 3.3.3 三路合并冲突解决协议

PPT 修改场景中存在一个经典问题：**用户反馈修改意见的同时，AI 可能也在基于其他用户的意见修改同一页**——即并发修改冲突。传统解决方案是使用 diff 工具展示变更让用户手动选择，但这与语音交互的流畅性相悖。本系统实现了**三路合并冲突解决协议**，将冲突解决融入对话式交互：

**三路合并的三方**：

| 版本 | 名称 | 来源 |
|---|---|---|
| **Base** | 基线快照 | 用户开始反馈时的页面状态 |
| **Current** | 当前版本 | 用户正在查看的页面（实时渲染） |
| **Incoming** | 新版本 | LLM 基于用户意图生成的修改 |

**合并策略矩阵**：

| 场景 | 策略 | 结果状态 |
|---|---|---|
| Base = Current | 直接采纳 Incoming | auto_resolved |
| Current = Incoming | 保持 Current（LLM 无改动） | auto_resolved |
| Incoming = Base | 保留 Current（其他来源已修改） | auto_resolved |
| 三者均不同 | 执行三路 Diff，检测冲突 | ask_human（挂起） |

**三路 Diff 算法**采用 LCS（最长公共子序列）算法计算行级差异：

1. 分别计算 Base → Current 和 Base → Incoming 的变更行集合
2. 检测两套变更行集合之间的重叠（行号重叠或内容规范化后相似）
3. 根据重叠程度分类：`content_conflict`（纯内容差异）、`structural_conflict`（行数变化 > 30%）、`mixed_conflict`（混合）、`general_conflict`（一般）
4. 若有冲突，生成用户友好的选择问题，用户通过 `ppt_mod` 回复

**超时与自动跳过**：冲突挂起后最多等待 45 秒（可重试 3 次），超时后自动跳过并处理 Pending 队列中的下一条反馈，确保对话不被长期阻塞。

### 3.3.4 打断恢复与不完整动作处理

流式输出中的打断恢复是语音交互系统的公认难题。本系统的解决方案包含以下步骤：

1. **打断检测**：用户 VAD 开始信号触发 `OnInterrupt()`
2. **流式取消**：取消当前的 Think Stream 和 Process Flow
3. **等待闭合**：最多等待 2 秒（每秒检查一次），让未闭合的 `@{...}` 协议块自然关闭
4. **不完整块裁剪**：通过 `TrimTrailingIncompleteAction` 算法删除尾部不完整的协议块
5. **轨迹保存**：将已完成的可见文本追加 `</interrupted>` 标记后写入对话历史

`TrimTrailingIncompleteAction` 算法通过计数 `{` 和 `}` 的嵌套深度，精确定位未闭合协议块的起始位置并截断。悬垂的 `@` 符号（单独出现而未跟随 `{`）也会被删除。下一轮推理时，LLM 看到 `</interrupted>` 标记即可理解之前的输出被打断，并基于此恢复上下文。

### 3.3.5 参考文献融合与意图解析

**参考文献融合（Ref Fusion）**：用户上传的参考文件（PDF、DOCX、PPTX、图片、视频）经过批量并发解析后，通过 LLM 定向抽取与当前 PPT 页面主题相关的原文片段。不同于传统的摘要生成，定向抽取**不总结、不改写**，保留原文准确性。PPTX 文件额外提取主题颜色、字体和版式作为样式指南，供 PPT 渲染时参考。

**意图解析（Intent Parser）**：采用 **LLM + 关键词双轨解析**。LLM 解析优先，输出结构化 JSON；若 LLM 不可用，关键词回退解析接管，支持多意图拆分（一条消息包含多个操作）、智能页面匹配（"这一页"、"第 2 页"自动识别）和操作类型检测（修改、删除、插入、重排）。

## 3.4 创新点总结

综合以上分析，本项目的核心创新点可以归纳为以下几个层次：

### 系统架构层面

| 创新点 | 技术方案 | 解决的问题 |
|---|---|---|
| 多 Goroutine 事件驱动 Pipeline | Channel 通信 + select 多路复用 | 各模块异步并发，最大化吞吐 |
| Idle Trigger 机制 | CAS 原子状态转换 + 自动唤醒 | 工具结果在用户未发言时主动触发播报 |
| 打断恢复机制 | 原始 token 缓冲 + 不完整块裁剪 + `</interrupted>` 标记 | 流式输出中的优雅打断与上下文恢复 |
| Channel Buffer 自适应 | 利用率反馈 + 动态扩缩容 | 运行时自适应内存分配 |

### 算法机制层面

| 创新点 | 技术方案 | 解决的问题 |
|---|---|---|
| 动态分层 System Prompt | 4 层架构，状态驱动更新 | 模型实时感知最新上下文 |
| 三路合并冲突解决 | LCS 行级 Diff + 分类 + 用户选择 | 并发修改冲突的对话式解决 |
| 记忆多维度召回 | 意图检测 + 加权评分 + 智能预算分配 | 精准匹配当前对话意图的记忆 |
| 偏好衰减机制 | 幂函数时间衰减 + 长/短期区分 | 避免过时偏好干扰当前决策 |
| KV Cache 预热 | ASR 增量触发 + token 丢弃 | 降低首 token 延迟 |
| RAG 混合检索 | Dense + BM25 + RRF + Jina Reranker | 兼顾语义相关性与关键词精确性 |

### 模型与数据层面

| 创新点 | 技术方案 | 解决的问题 |
|---|---|---|
| 打断检测 LoRA 微调 | Qwen3-0.6B + LoRA | 精准检测用户打断意图 |
| LLM Tool Calling 冲突检测 | 结构化 tool 输出 | 泛化能力极强的动态冲突判断 |
| LLM 意图精化 | 查询扩写为检索关键词 | 模糊自然语言到精准检索的转换 |

本项目的整体设计理念可以概括为：**以语音交互为核心驱动，以事件驱动为架构基础，以自适应为优化手段，以对话式解决为用户体验**。系统的每一个模块都经过精心设计，在保证高并发、低延迟的技术要求的同时，最大化用户通过自然语音完成复杂教学课件制作的体验流畅性。
