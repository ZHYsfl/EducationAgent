# 接口规范差异分析报告

> 对照文档：`系统接口规范.md` v1.0（2026-03-19）  
> 分析时间：2026-04-05  
> 最后更新：2026-04-05  
> 分析范围：`voice_agent/` 模块全部 Go 源码  
> 结论：共发现 **19 处**不一致。已修复 S1、S3、S4、S7、M3、L1（共 6 处），剩余 13 处待处理。

---

## 严重问题（影响跨模块联调）

### S2. ContextMessage.Source 字段名错误（§1.2）

**规范要求：**
```go
type ContextMessage struct {
    Source string `json:"source"` // 消息来源模块
    ...
}
```

**实际代码（`internal/types/types.go:22`）：**
```go
ActionType string `json:"action_type"` // 字段名和 JSON key 均不符
```

**影响：** 所有依赖 `source` 字段做路由/过滤的下游模块（PPT Agent、KB Service 等）无法正确识别消息来源。

---

### S5. asyncQuery 函数未实现（§1.4）

**规范要求：** 定义 `asyncQuery(ctx, source, msgType, queryFn)` 统一异步查询模式。

**实际代码：** 全 codebase 无 `asyncQuery` 函数。各处异步查询逻辑分散实现，未遵循规范定义的统一模式。

**影响：** 异步上下文注入的行为不可预期，`msgType` 传递不规范。

---

### S6. ~~前端从未收到 `conflict_ask` 消息~~ → 已确认实现，但顺序错误（§2.1.2、§3.4）

> **注：** 经代码复核，`conflict_ask` WS 消息**确实已发送**（`bus.go:50-57`），原报告结论有误。此问题降级为轻微（见 L4 修订）。

---

### S8. 需求收集状态机不完整（§2.4.2）

**规范要求状态流转：** `collecting → confirming → confirmed → generating`

**实际代码（`pipeline_post.go:69-73`）：**
```go
if req.IsReadyForConfirm() {
    req.Status = "ready"    // 用 "ready" 代替规范的 "confirming"
} else {
    req.Status = "collecting"
}
```

- `confirming` 状态从未设置（用 `"ready"` 替代）
- `confirmed` 状态无对应处理逻辑
- `generating` 状态不由需求状态机设置

**影响：** 需求确认流程无法正常走完。

## 中等问题（功能不完整或字段缺失）

### M1. TaskRequirements 字段问题（§2.4.1）

**两处不符：**

1. **多出 `Subject` 字段**（`requirements.go:16`）：规范 §2.4.1 的结构体定义中无此字段。

2. **`GetMissingFields()` 将可选字段当必填**（`requirements.go:77-119`）：

| 字段 | 规范优先级 | 代码处理 |
|---|---|---|
| `subject` | 规范中不存在 | 必填 ✗ |
| `total_pages` | P2 可选 | 必填 ✗ |
| `key_difficulties` | P1 可选 | 必填 ✗ |
| `duration` | P1 可选 | 必填 ✗ |
| `global_style` | P1 可选 | 必填 ✗ |
| `interaction_design` | P2 可选 | 必填 ✗ |
| `output_formats` | P2 可选 | 必填 ✗ |

`IsReadyForConfirm()` 依赖 `GetMissingFields()`，导致系统永远不会进入确认阶段，除非用户填完所有 P1/P2 字段。

---

### M2. `ToPPTInitRequest` 方法缺失（§2.4.5）

**规范要求：** `TaskRequirements` 上定义 `ToPPTInitRequest() PPTInitRequest` 方法。

**实际代码：** 该方法不存在。转换逻辑散落在 `internal/executor/ppt.go:91` 内联实现，未封装为规范定义的方法。

---

### M4. `task_init` 入站消息未处理（§2.1.2）

**规范要求：** 前端可发送 `task_init` 消息（含 `topic`, `description`, `total_pages`, `audience`, `global_style`）通过表单直接初始化 PPT 任务。

**实际代码（`session_ws.go:11-23`）：** switch 中无 `case "task_init"`，消息被静默丢弃。

---

### M5. `requirements_confirm` 入站消息未处理（§2.4.6）

**规范要求：** 前端可发送 `requirements_confirm` 消息（含 `confirmed: bool`, `modifications: string`）确认或修改需求。

**实际代码（`session_ws.go:11-23`）：** switch 中无 `case "requirements_confirm"`，消息被静默丢弃。

---

### M6. MemoryRecallRequest / MemoryExtractRequest 字段缺失（§5.3.1、§5.3.2）

**MemoryRecallRequest（`types.go:110-115`）缺少：**
`IncludeCards`, `IncludeEvidence`, `IncludeHints`, `IncludeInactive`, `EvidenceDetailLevel`, `HydrateEvidence`

**MemoryExtractRequest（`types.go:189-193`）缺少：**
`WindowTurns`, `EnableCardUpsert`, `EnableDialogueIndex`

**影响：** 无法使用规范定义的高级记忆检索/提取功能（卡片、证据片段、主动提示等）。

---

### M7. `prefillFromMemory` 函数缺失（§2.4.7）

**规范要求：** 需求收集开始时，从 Memory Service 预填已知用户偏好到 `TaskRequirements`。

**实际代码：** 全 codebase 无 `prefillFromMemory` 函数。记忆召回结果仅用于 system prompt 上下文，不会预填需求字段。

---

## 轻微问题（字段缺失或类型偏差，不影响主流程）

### L2. MemoryExtractResponse 类型错误（§5.3.1）

**规范：** `ExtractedFacts []MemoryEntry`, `ExtractedPreferences []MemoryEntry`  
**代码（`types.go:200-204`）：** `ExtractedFacts []string`, `ExtractedPreferences []string`

---

### L3. WorkingMemory.ExtractedElements 类型错误（§5.2.1）

**规范：** `ExtractedElements TeachingElements`（强类型结构体）  
**代码（`types.go:135`）：** `ExtractedElements map[string]any`（弱类型）

`WorkingMemorySaveRequest`（`types.go:210`）同样使用 `map[string]any`。

---

### L4. highPriorityListener 中 conflict_ask 发送顺序与 AddPendingQuestion 时机问题（§1.3.3）

**规范要求顺序：** ① TTS播报 → ② 记录 context_id（AddPendingQuestion） → ③ 发送 conflict_ask

**实际代码（`bus.go:49-110`）顺序：** ① 发送 conflict_ask → ② TTS播报 → ③ TTS 完成后才调用 AddPendingQuestion

两处偏差：
1. `conflict_ask` 在 TTS 之前发出（规范要求在 TTS 之后）
2. 若 TTS 被用户打断（`ctx.Err() != nil`），进入重试分支，`AddPendingQuestion` 不会被调用，冲突问题在重试期间无法被 `ResolvePendingQuestion` 找到。

---

### L5. PPTMessageRequest 包含规范外字段（§3.4）

**规范定义字段：** `TaskID`, `PageID`, `Priority`, `ContextID`, `TTSText`, `MsgType`（共 6 个）

**代码额外字段（`messages.go:78-86`）：** `RenderURL`, `PageIndex`, `DownloadURL`, `Format`, `Progress`, `Status`, `PageOrder`, `PagesInfo`, `ErrorCode`（共 9 个）

这些字段是对规范的扩展，用于复用该结构体传递渲染/导出状态，未在规范中记录。
